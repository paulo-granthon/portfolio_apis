[~50s] A arquitetura foi planejada em três serviços independentes, todos a partir do usuário. O usuário acessa o frontend React, que consome as duas APIs por REST. A API Python/Flask cuida dos dados de yield e usa MongoDB, mas também acessa o PostgreSQL principal — não só o Mongo. O serviço de autenticação em Rust usa dois bancos PostgreSQL separados: o principal, que é compartilhado com o Flask, e um banco dedicado exclusivamente para chaves criptográficas, o api6_keys — propositalmente isolado do resto. Toda a infraestrutura foi orquestrada com um monorepo Nx, Docker Compose sobe tudo com um comando, o GitHub Actions cuida do CI/CD, Poetry gerencia as dependências do Python e o SeaORM é o ORM assíncrono do lado Rust.
// 03 · api6 · destaque técnico

serviço de auth em Rust

  • Microsserviço construído do zero com Actix-web
  • JWT com assinatura criptográfica para geração e validação de tokens
  • Revogação de tokens via tabela revoked_tokens no PostgreSQL
  • Middleware de proteção de rotas integrado
  • Documentação OpenAPI / Swagger completa
  • Login de usuário inativo bloqueado na camada de serviço
  • Serve um botão HTML embarcável e a página de login (template) já configurada com as informações necessárias; ao autorizar, retorna um token para o cliente externo — que precisa estar previamente cadastrado no sistema

por que Rust?

Segurança de memória em tempo de compilação, sem garbage collector. Ideal para serviços críticos onde bugs de memória são inaceitáveis.

Actix-web JWT SeaORM Segurança
[~60s] O coração técnico do api6 foi o serviço de autenticação em Rust. Concebi e implementei do zero com Actix-web: geração e validação de tokens JWT com assinatura criptográfica, revogação de tokens persistida em banco (não só expiração), middleware que protege rotas automaticamente e documentação OpenAPI gerada. Além disso, o serviço serve um botão HTML que qualquer cliente externo pode embutir na própria página — esse botão abre a página de login (um template já servido pelo Rust, com todas as informações pré-requisito configuradas), o usuário autoriza, e o sistema retorna um token que o cliente externo usa para buscar os dados. Importante: só clientes externos previamente cadastrados podem usar esse fluxo. A escolha de Rust foi estratégica: um serviço de autenticação é crítico — qualquer bug de memória pode gerar vulnerabilidades de segurança. Rust elimina essa categoria inteira de problemas em tempo de compilação.
// 03 · api6 · destaque técnico

criptografia & portabilidade

criptografia Fernet de campos sensíveis

  • Módulo fernet em Rust cifrando campos de usuário
  • Banco api6_keys separado para isolar chaves criptográficas
  • Criptografia aplicada já na carga dos seeds — nenhum dado sensível em texto puro

clientes externos e portabilidade de dados

  • Tabela external_clients com chaves únicas por entidade
  • JWT ampliado para suportar dupla entidade (usuário + cliente externo)
  • Fluxo em 3 etapas: botão → autorização → token → dados
  • App de exemplo simulando o cliente externo
[~60s] Além do JWT, implementei criptografia Fernet para campos sensíveis dos usuários. A decisão de separar as chaves criptográficas num banco dedicado — o api6_keys — foi deliberada: compromisso do banco principal não expõe as chaves. Implementei também o fluxo completo de clientes externos e portabilidade de dados — um usuário pode autorizar um sistema externo a acessar seus dados em três etapas, com JWT que carrega dupla entidade. Esse fluxo exigiu integrar Rust, banco e segurança de ponta a ponta.
